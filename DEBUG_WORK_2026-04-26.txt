Main bugs found
- WebRTC fallback was too aggressive. A 10 second timeout could switch transfers to Socket.IO relay, which is much slower for real files.
- Sender and receiver could join the same room twice, causing duplicate WebRTC offers.
- Duplicate offers caused this browser error:
  Failed to execute 'createAnswer' on 'RTCPeerConnection'
- Large transfers could stall around 16 MB because the sender filled the browser DataChannel buffer too far ahead of the receiver.
- Next.js dev mode could pick the wrong Turbopack root because the repository has root and client lockfiles.

Main fixes made
- Added shared client URL config in client/src/lib/config.js.
- Added duplicate join and duplicate handshake guards in client/src/lib/transferManager.js.
- Made the signaling server idempotent for same socket room rejoins in server/signaling.js.
- Reworked WebRTC sending flow in client/src/lib/webrtc.js:
  - Receiver sends chunk acknowledgements.
  - Sender uses a bounded send window.
  - Browser buffer limit was reduced to avoid stall-prone bursts.
  - Completion waits for receiver acknowledgement instead of only local buffer drain.
- Kept Socket.IO relay only as a small-file fallback, not as the normal large-file path.
- Added socket relay acknowledgement/backpressure for fallback mode.
- Pinned Next.js Turbopack root in client/next.config.mjs.
- Added client/vercel.json for Vercel deployment defaults.
- Added DEPLOYMENT.md with Vercel and Render setup instructions.
- Updated .env.example with production examples.
